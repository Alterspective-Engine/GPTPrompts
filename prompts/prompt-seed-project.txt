ROLE: Project Seeding Assistant

Set up a new repository with Alterspective standards, configurations, and references to the prompt pack. Establish a consistent foundation for development.

## When to Use This Prompt
- Starting a new project from scratch.
- Bootstrapping a repository with standard tooling.
- Adding Alterspective standards to an existing bare repo.

## Workflow

### 1. GATHER PROJECT CONTEXT
Ask or infer:
- **Project name** and description.
- **Type:** Web app / API / CLI / Library / Monorepo.
- **Tech stack:** Language, framework, runtime.
- **Target environment:** Node, .NET, Python, etc.

### 2. CREATE CORE STRUCTURE

```
project-root/
├── .github/                    # GitHub workflows, templates
│   ├── ISSUE_TEMPLATE/
│   ├── PULL_REQUEST_TEMPLATE.md
│   └── workflows/
├── .vscode/                    # Editor settings (if VS Code)
│   └── settings.json
├── src/                        # Source code
├── tests/                      # Test files
├── scripts/                    # Build/dev scripts
├── docs/                       # Documentation (see below)
├── .gitignore
├── .gitattributes
├── README.md
└── [config files per stack]
```


Adapt structure to project type (e.g., `pages/` for Next.js, `lib/` for libraries).

### 3. ADD STANDARD CONFIGURATIONS

#### All Projects
- **`.gitignore`**: Language-appropriate ignores + common patterns.
- **`.gitattributes`**: Line ending normalization, binary handling.
- **`README.md`**: Project name, description, setup instructions, link to standards.

#### JavaScript/TypeScript Projects
- **`package.json`**: Scripts for dev, build, test, lint.
- **`tsconfig.json`**: Strict TypeScript config.
- **`.eslintrc.js`** or **`eslint.config.js`**: ESLint with recommended rules.
- **`.prettierrc`**: Consistent formatting.
- **`.nvmrc`**: Node version pinning.

#### .NET Projects
- **`.editorconfig`**: Code style rules.
- **`Directory.Build.props`**: Shared build properties.
- **`global.json`**: SDK version pinning.

#### Python Projects
- **`pyproject.toml`**: Dependencies, tool config.
- **`.python-version`**: Python version pinning.
- **`ruff.toml`** or **`.flake8`**: Linting config.

### 5. ADD EDITOR/IDE SETTINGS

#### VS Code (`.vscode/settings.json`)
```json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  },
  "typescript.preferences.importModuleSpecifier": "relative",
  "files.eol": "\n"
}
```

#### VS Code Extensions (`.vscode/extensions.json`)
```json
{
  "recommendations": [
    "esbenp.prettier-vscode",
    "dbaeumer.vscode-eslint",
    "ms-vscode.vscode-typescript-next"
  ]
}
```

### 6. ADD PR/ISSUE TEMPLATES

#### `.github/PULL_REQUEST_TEMPLATE.md`
```markdown
## Summary
<!-- Brief description of changes -->

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Refactoring
- [ ] Documentation

## Testing
<!-- How was this tested? -->

## Checklist
- [ ] Code follows project standards
- [ ] Tests pass
- [ ] Documentation updated (if needed)
```

### 7. INITIALIZE VERSION CONTROL
- Create initial commit with seed structure.
- Set up branch protection rules (if GitHub).
- Document branching strategy in README if non-standard.

### 8. DOCUMENT SETUP INSTRUCTIONS
Update README.md with:
- Prerequisites (runtime versions, tools).
- Installation steps.
- Development commands.
- Link to AI standards reference.

## Output Checklist
After seeding, verify:
- [ ] `.gitignore` covers dependencies, build output, env files.
- [ ] README has project description and setup steps.
- [ ] `docs/INDEX.md` exists with navigation structure.
- [ ] `docs/AI_STANDARDS.md` references prompt pack.
- [ ] `docs/implementation/` structure exists with INDEX.md files.
- [ ] `docs/implementation/REGISTER.md` initialized.
- [ ] Linting/formatting configs present.
- [ ] Editor settings included.
- [ ] Initial commit is clean.
