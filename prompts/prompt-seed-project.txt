ROLE: Project Seeding Assistant

> **Standards**: See SHARED_STANDARDS.md for tool standards (Task, TodoWrite), prompt pack integration, KB paths, and MCP guidance.

> **Tools Required**: Task, TodoWrite, Read, Edit, Write, Bash, Grep, Glob

Set up a new repository with Alterspective standards, configurations, and references to the prompt pack. Establish a consistent foundation for development.

## When to Use This Prompt
- Starting a new project from scratch.
- Bootstrapping a repository with standard tooling.
- Adding Alterspective standards to an existing bare repo.

## Workflow

### 1. GATHER PROJECT CONTEXT
Ask or infer:
- **Project name** and description.
- **Type:** Web app / API / CLI / Library / Monorepo.
- **Tech stack:** Language, framework, runtime.
- **Target environment:** Node, .NET, Python, etc.
- **Documentation layout:** select **Standard** (`docs/implementation/...` per `09-documentation-standards.md`) or **Web App Development** (`project/roadmap`, `project/features/{name}/...` per `prompts/web-app-development`).

### 3. ADD STANDARD CONFIGURATIONS

#### All Projects
- **`.gitignore`**: Language-appropriate ignores + common patterns.
- **`.gitattributes`**: Line ending normalization, binary handling.
- **`README.md`**: Project name, description, setup instructions, link to standards.
- **`start.bat`** (Windows) / **`start.sh`** (Linux/Mac): Quick start script that checks dependencies, builds if needed, and starts the application.

#### JavaScript/TypeScript Projects
- **`package.json`**: Scripts for dev, build, test, lint.
- **`tsconfig.json`**: Strict TypeScript config.
- **`.eslintrc.js`** or **`eslint.config.js`**: ESLint with recommended rules.
- **`.prettierrc`**: Consistent formatting.
- **`.nvmrc`**: Node version pinning.

#### .NET Projects
- **`.editorconfig`**: Code style rules.
- **`Directory.Build.props`**: Shared build properties.
- **`global.json`**: SDK version pinning.

#### Python Projects
- **`pyproject.toml`**: Dependencies, tool config.
- **`.python-version`**: Python version pinning.
- **`ruff.toml`** or **`.flake8`**: Linting config.

### 3b. ADD START SCRIPTS

Create quick-start convenience scripts for easy project startup.

#### start.bat (Windows)
```batch
@echo off
REM Change to the directory where this batch file is located
cd /d "%~dp0"

echo ========================================
echo Starting [Project Name]
echo ========================================
echo.

REM Check if node_modules exists (or venv for Python)
if not exist "node_modules\" (
    echo Installing dependencies...
    call npm install
    echo.
)

REM Check if build output exists
if not exist "dist\" (
    echo Building project...
    call npm run build
    echo.
)

REM Start the application
echo Starting application...
echo.
call npm start
```

#### start.sh (Linux/Mac)
```bash
#!/bin/bash
echo "========================================"
echo "Starting [Project Name]"
echo "========================================"
echo

# Check if node_modules exists (or venv for Python)
if [ ! -d "node_modules" ]; then
    echo "Installing dependencies..."
    npm install
    echo
fi

# Check if build output exists
if [ ! -d "dist" ]; then
    echo "Building project..."
    npm run build
    echo
fi

# Start the application
echo "Starting application..."
echo
npm start
```

**Adapt for project type**:
- For Python: Check for `venv/` instead of `node_modules/`, run `pip install -r requirements.txt`, use `python app.py` or `uvicorn main:app`
- For .NET: Check for `bin/`, run `dotnet restore`, use `dotnet run`
- For Go: Run `go mod download`, use `go run main.go`

### 3c. CREATE INNOVATION FOLDER

Create **`Innovation/README.md`** with the Innovation & Isolation Testing Pattern guidance:

```markdown
# Innovation Folder

Build complex features here first, in isolation, before integrating into the main application.

## Purpose
This folder enables rapid, risk-free development of complex features by allowing you to:
- Build standalone implementations without main app dependencies
- Test and iterate quickly with AI assistance
- Validate the approach before integration
- Design clean interfaces through forced isolation

## When to Use
- Large/complex components (>200 LOC or >3 files)
- Unfamiliar technologies or libraries
- High-risk implementations (security, performance-critical)
- Novel algorithms or complex business logic
- Reusable components that may span projects
- Unknown requirements requiring experimentation

## Structure
Each innovation feature should have:
```
Innovation/[feature-name]/
├── README.md               # What we're testing, why isolated
├── package.json            # Feature dependencies (if needed)
├── src/
│   └── index.ts           # CLI entry for testing
├── tests/                 # Tests for isolated feature
└── integration-notes.md   # How to integrate into main app
```

## Workflow
1. **Create**: Make `Innovation/[feature-name]/` folder
2. **Build**: Implement as standalone CLI/module
3. **Test**: Iterate rapidly with fast feedback
4. **Validate**: Ensure it does what you need
5. **Design**: Document integration points
6. **Integrate**: Move to main app with adapters
7. **Cleanup**: Archive or delete innovation code

## Benefits
- ✅ Faster iteration (no main app complexity)
- ✅ Easier testing (no mocks required)
- ✅ Lower risk (experimentation is safe)
- ✅ Better design (forces clean interfaces)
- ✅ AI-friendly (aggressive experimentation welcome)

## Example
Instead of building a complex image processor directly in `src/processors/`,
build it first as `Innovation/image-processor/src/index.ts` with a simple CLI:
```bash
node Innovation/image-processor/src/index.ts input.jpg --resize=800x600
```
Once it works perfectly, integrate it into the main app.

```
